# 在 IM 消息中实现大模型多模态交互

最近给 ClaudeTalk 加了多模态支持，让飞书机器人能接收图片和文件，并传给 Claude 进行分析。

这篇文章记录一下整个实现过程，以及中间踩的坑。

项目地址：https://github.com/suyin58/claudetalk

---

## 为什么要做多模态

在 IM 里跟 AI 交互，最自然的诉求就是：**发张图，让 AI 帮我看看**。

比如：
- 截一张报错截图，问 AI 怎么解决
- 发一张设计稿，让 AI 帮忙写实现代码
- 发一份 PDF 文档，让 AI 帮忙总结要点

这些场景在纯文字对话里都很别扭——你得把图片描述成文字，或者把文档内容复制粘贴进去。有了多模态支持，直接发就行了。

---

## IM 平台的多模态消息结构

不同 IM 平台对图片/文件消息的处理方式不太一样，但核心思路是一致的：**平台不直接在消息体里传二进制内容，而是给你一个下载凭证，让你自己去下载**。

### 飞书的消息结构

飞书的消息类型（`msg_type`）有这几种：

```json
// 图片消息
{
  "msg_type": "image",
  "content": "{\"image_key\": \"img_xxx\"}"
}

// 文件消息
{
  "msg_type": "file",
  "content": "{\"file_key\": \"75235e0c-xxx\", \"file_name\": \"需求文档.pdf\"}"
}

// 富文本（文字 + 图片混排）
{
  "msg_type": "post",
  "content": "{\"content\": [[{\"tag\": \"text\", \"text\": \"看这张图\"}, {\"tag\": \"img\", \"image_key\": \"img_xxx\"}]]}"
}
```

关键字段：
- **`image_key`**：图片的唯一标识，用于下载图片
- **`file_key`**：文件的唯一标识，用于下载文件
- **`file_name`**：文件的原始名称（这个很重要，后面会说）

### 钉钉的消息结构

钉钉的图片消息 `msgtype` 是 `picture`（注意不是 `image`），用 `downloadCode` 作为下载凭证：

```json
{
  "msgtype": "picture",
  "content": {
    "downloadCode": "mIofN681YE3f/+m+..."
  }
}
```

钉钉的 `downloadCode` 是一个加密的临时凭证，**有时效性**，这和飞书的 `image_key` / `file_key` 是全局唯一且持久有效的不同。这个差异对缓存策略影响很大。

---

## 下载图片/文件

拿到下载凭证后，调飞书的资源下载接口：

```
GET /im/v1/messages/{message_id}/resources/{image_key}?type=image
GET /im/v1/messages/{message_id}/resources/{file_key}?type=file
```

注意：**必须同时传 `message_id` 和资源 key**，只传 key 是不够的。这是我踩的第一个坑——最开始用的是 `/im/v1/images/{imageKey}` 接口，一直 HTTP 400，后来才发现要用消息资源接口。

---

## 缓存策略：image_key 是全局唯一的

飞书的 `image_key` 和 `file_key` 是**全局唯一**的，同一张图片无论在哪条消息里出现，`image_key` 都一样。

这个特性让缓存变得非常简单：**以 `image_key` / `file_key` 为文件名，下载一次永久复用**。

```
workDir/
└── .claudetalk/
    └── feishu/
        ├── images/
        │   └── {image_key}.jpg      ← 图片缓存
        └── files/
            └── {file_key}.{ext}     ← 文件缓存（保留原始后缀）
```

实现逻辑很简单：

```typescript
async downloadImage(imageKey: string, messageId: string): Promise<string | null> {
  const localPath = path.join(imageDir, `${safeImageKey}.jpg`)

  // 缓存命中，直接返回
  if (fs.existsSync(localPath)) {
    return localPath
  }

  // 下载并保存
  const response = await fetch(
    `${FEISHU_API_BASE}/im/v1/messages/${messageId}/resources/${imageKey}?type=image`,
    { headers: { Authorization: `Bearer ${accessToken}` } }
  )
  const arrayBuffer = await response.arrayBuffer()
  fs.writeFileSync(localPath, Buffer.from(arrayBuffer))
  return localPath
}
```

这个缓存策略的好处是：**实时消息和历史消息共用同一份缓存**。同一张图片在历史消息里出现过，实时消息里再次出现时直接命中缓存，不需要重复下载。

## 怎么把文件传给 Claude

Claude Code 本身支持读取本地文件（通过 Read 工具），所以不需要把图片/文件内容直接塞进 prompt，只需要**告诉 Claude 文件在哪里**就行了：

```
用户消息：帮我分析这张图
[图片: /Users/xxx/.claudetalk/feishu/images/img_xxx.jpg]
```

Claude 看到这个提示后，会自动调用 Read 工具读取图片内容进行分析。

文件消息还需要附带原始文件名，让 Claude 知道这是什么文件：

```
用户消息：帮我总结这份文档
[文件: /Users/xxx/.claudetalk/feishu/files/75235e0c.pdf (需求文档.pdf)]
```

为什么要单独传文件名？因为本地保存的文件名是 `{file_key}.{ext}`，Claude 从路径里看不出这是什么文件，加上原始文件名后上下文更清晰。

## 纯图片/纯文件消息的处理：单聊和群聊不一样

用户直接发一张图片，不带任何文字，这时候应该怎么处理？

**单聊和群聊的操作路径完全不同，处理方式也不一样。**

### 单聊：缓存图片，等待指令

单聊里，用户发图片，机器人能直接收到这条消息。但如果直接调 Claude，Claude 会收到一个空消息 + 图片路径，它不知道你想让它做什么，只能随便说点什么。

更好的做法是：**缓存图片，等待用户发指令**。

```
用户：[发了一张图片]
机器人：📎 已收到（共 1 个文件/图片），请继续发送指令。

用户：帮我分析这张图里的数据
机器人：[Claude 收到：帮我分析这张图里的数据 + 图片路径，开始分析]
```

实现上用一个 Map 缓存待处理的图片路径：

```typescript
// key: `${conversationId}:${senderId}`，不同用户的缓存互不干扰
private pendingImages = new Map<string, string[]>()

// 纯图片消息：缓存，回复提示
if (hasImages && !hasText) {
  const cached = this.pendingImages.get(pendingKey) ?? []
  this.pendingImages.set(pendingKey, [...cached, ...imagePaths])
  await this.sendMessage(conversationId, `📎 已收到（共 ${cached.length + imagePaths.length} 个文件/图片），请继续发送指令。`)
  return
}

// 有文字时：合并缓存的图片，一起传给 Claude
const allImagePaths = [...(this.pendingImages.get(pendingKey) ?? []), ...imagePaths]
this.pendingImages.delete(pendingKey)
```

支持连续发多张图片再发指令，图片会按顺序全部传给 Claude。

### 群聊：通过历史消息获取图片

群聊里，用户发图片**不会 @ 机器人**，机器人根本收不到这条消息，`pendingImages` 缓存机制在群聊里完全无效。

群聊的正确操作路径是：

```
用户：[发了一张图片]（机器人收不到）

用户：@机器人 帮我分析这张图里的数据
机器人：拉取历史消息 → 发现上一条是图片消息 → 下载图片
       → Claude 收到：帮我分析这张图里的数据 + 图片路径，开始分析
```

机器人被 @ 时，会拉取最近 N 条历史消息作为上下文。历史消息里的图片和文件同样需要下载并传给 Claude，处理逻辑和实时消息一致，同样走缓存复用。

飞书历史消息 API 有个坑：**字段名是 `msg_type`，不是 `message_type`**。这个坑让我调试了很久，历史消息里的图片一直显示原始 JSON 而不是图片路径。

```typescript
// 历史消息 API 响应
interface FeishuHistoryMessage {
  message_id: string
  msg_type: string        // ← 注意是 msg_type，不是 message_type
  body: { content: string }
  // ...
}
```

## 各消息类型的支持情况

| 消息类型 | 飞书实时消息 | 飞书历史消息 | 钉钉 |
|---------|------------|------------|------|
| 文字 | ✅ | ✅ | ✅ |
| 图片 | ✅ | ✅ | ❌ |
| 富文本（文字+图片） | ✅ | ✅ | ❌ |
| 文件 | ✅ | ✅ | ❌ |
| 语音、视频 | ❌ 回复提示 | 标注类型 | ❌ 回复提示 |

钉钉不支持多模态的原因：
1. `downloadCode` 有时效性，无法做持久缓存
2. 钉钉群聊中机器人不支持接收语音、文件、视频类型
3. 钉钉没有历史消息 API，无法拉取群聊上下文

## 几个值得注意的细节

**1. 文件名要保留原始后缀**

本地保存文件时，从 `file_name` 里提取后缀，保留到本地文件名里。这样 Claude 能通过文件后缀判断文件类型，读取时也能正确处理。

**2. file_key 里可能有特殊字符**

`file_key` 格式类似 UUID（`75235e0c-4f92-430a-a99b-8446610223cg`），用作文件名时需要把 `-` 以外的特殊字符替换掉：

```typescript
const safeFileKey = fileKey.replace(/[^a-zA-Z0-9_-]/g, '_')
```

**3. 群聊中不同用户的缓存要隔离**

`pendingImages` 的 key 是 `${conversationId}:${senderId}`，不同用户发的图片不会混在一起。

**4. 文件名信息的传递**

`downloadFile` 方法只返回本地路径，原始文件名在调用方已知（从消息体里解析出来）。拼提示文本时，用 `|` 分隔符把文件名携带进 `imagePaths` 数组，最终拼成 `[文件: /path/xxx.pdf (需求文档.pdf)]` 的格式传给 Claude。

## 总结

在 IM 里实现多模态交互，核心链路其实不复杂：

```
IM 消息事件
  → 解析消息类型（text / image / file / post）
  → 下载图片/文件到本地（以 key 为唯一标识，缓存复用）
  → 拼接提示文本（文字 + 图片/文件路径 + 原始文件名）
  → 传给 Claude，Claude 用 Read 工具读取文件内容
  → 回复用户
```

最麻烦的地方不是下载本身，而是各种边界情况：纯图片消息怎么处理、历史消息里的图片怎么处理、文件名怎么传递给 Claude、不同用户的缓存怎么隔离。

这些细节处理好了，用起来就很自然——在飞书群里发张截图，@ 一下机器人，它就能帮你分析了。
